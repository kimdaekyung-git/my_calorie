## 1. 문제 정의

### 1.1 현재 상황 (As-Is)
노션이나 메모장에 먹은 것을 직접 적고 있지만, 구조화가 안 되어 나중에 돌아보면 무슨 소리인지 모르겠는 상황. 마치 일기장에 "점심 맛있었음"이라고만 적어놓은 것처럼, 정보로서의 가치가 없는 기록이 쌓이고 있습니다.

### 1.2 목표 상황 (To-Be)
수시로 먹을 때마다 3초 안에 기록을 남기고, 일주일이 지나면 "아, 내가 이런 패턴으로 먹고 있구나"를 한눈에 파악할 수 있는 상태. 기록이 쌓일수록 내 식단의 지도가 그려지는 느낌.

### 1.3 왜 지금인가? (Why Now)
개인적으로 식단 관리가 필요한 시점이고, 기존 앱들은 너무 복잡해서 쓰다가 포기하게 됨. 내가 직접 쓸 간단한 도구를 만들어보고 싶은 동기.

## 2. 사용자 페르소나

### 2.1 주요 페르소나

| 항목 | 내용 |
|------|------|
| 이름/별칭 | "다이어트 중인 나" |
| 나이/직업 | 식단 관리에 관심 있는 사람 |
| 핵심 니즈 | 먹은 것을 귀찮지 않게 기록하고 패턴 파악 |
| 불편 포인트 | 기존 방식(메모장)은 구조화 안 됨, 기존 앱은 복잡함 |
| 사용 상황 | 먹을 때마다 수시로 스마트폰에서 기록 |

### 2.2 페르소나의 하루 (시나리오)
아침에 일어나 간단히 시리얼을 먹고, 헬스메이트를 열어 "시리얼 + 우유"를 빠르게 검색해서 탭 한 번으로 기록한다. 점심에는 회사 근처 식당에서 김치찌개를 먹고, 자리에 앉자마자 "김치찌개"를 검색해 기록. 저녁에는 샐러드를 먹고 기록한다. 일주일이 지나면 주간 리포트를 열어 "이번 주 총 칼로리"와 "자주 먹은 음식"을 확인한다.

## 3. 사용자 스토리

### FEAT-0: 공통 흐름 (온보딩)
- [ ] 사용자로서, 앱을 처음 열면 바로 사용할 수 있어야 한다. 왜냐하면 로컬 저장 방식이라 별도 가입이 필요 없으니까.
- [ ] 사용자로서, 간단한 튜토리얼을 볼 수 있어야 한다. 왜냐하면 핵심 기능을 빠르게 이해해야 하니까.

### FEAT-1: 간편 식단 기록 (MVP)
- [ ] 다이어트 중인 사용자로서, 음식 이름을 검색하면 칼로리 정보가 자동으로 채워져야 한다. 왜냐하면 직접 칼로리를 입력하는 건 너무 귀찮으니까.
- [ ] 다이어트 중인 사용자로서, 최근에 기록한 음식을 탭 한 번으로 다시 기록할 수 있어야 한다. 왜냐하면 매일 비슷한 것을 먹으니까.
- [ ] 다이어트 중인 사용자로서, 아침/점심/저녁/간식으로 구분해서 기록할 수 있어야 한다. 왜냐하면 끼니별 패턴을 알고 싶으니까.
- [ ] 다이어트 중인 사용자로서, 오늘 먹은 것의 총 칼로리를 한눈에 볼 수 있어야 한다. 왜냐하면 하루 목표 대비 현재 상태를 알아야 하니까.

### FEAT-2: 칼로리 계산 (v2 이후)
- [ ] 사용자로서, 음식별 영양소(탄수화물/단백질/지방)를 확인할 수 있어야 한다.
- [ ] 사용자로서, 하루 목표 칼로리를 설정하고 현재 진행률을 볼 수 있어야 한다.

### FEAT-3: 주간 리포트 (v2 이후)
- [ ] 사용자로서, 일주일간 식단 패턴을 차트로 확인할 수 있어야 한다.
- [ ] 사용자로서, 자주 먹는 음식 TOP 5를 볼 수 있어야 한다.

## 4. 성공 지표

### 4.1 노스스타 지표 (결과 지표)
| 지표명 | 정의 | 목표치 | 측정 주기 |
|--------|------|--------|----------|
| 매일 3끼 기록률 | 하루 3끼 모두 기록한 날의 비율 | 50% 이상 | 주간 |

### 4.2 입력 지표 (행동 지표)
| 지표명 | 정의 | 목표치 |
|--------|------|--------|
| 일일 기록 횟수 | 하루에 식단을 기록하는 횟수 | 3회 이상 |
| 주간 연속 기록 | 연속으로 기록한 일수 | 5일 이상 |

## 5. 범위 관리

### 5.1 MVP (반드시 포함)
- FEAT-1: 간편 식단 기록 (음식 검색 + 칼로리 자동 입력 + 끼니별 기록)
- FEAT-0: 기본 온보딩 (가입 없이 바로 시작)

### 5.2 옵션 (시간 허용 시)
- 오늘 총 칼로리 요약 표시
- 최근 기록 재사용 기능

### 5.3 Non-goals (명시적 제외)
- 운동 기록 기능 - 식단에 집중하기 위해
- 소셜/공유 기능 - 개인 사용 목적이므로 불필요
- 회원가입/로그인 - 로컬 저장이라 불필요
- 다크 모드 - MVP에서는 라이트 모드만
- 사진으로 음식 인식 - 복잡도 높음, v2 이후 고려

## 6. 가정과 리스크

### 6.1 핵심 가정
| # | 가정 | 검증 방법 |
|---|------|----------|
| 1 | 음식 검색으로 입력하면 귀찮지 않을 것이다 | 1주일 직접 사용 후 기록 지속 여부 확인 |
| 2 | 칼로리 정보가 있으면 식단 관리 동기가 유지된다 | 주간 리포트 조회 빈도 확인 |
| 3 | 로컬 저장으로 충분하다 (기기 변경 걱정 없음) | 1달 사용 후 클라우드 필요성 평가 |

### 6.2 Top 5 리스크 보드

| # | 가정 | 리스크 | 완화 | 트리거 | 대응 |
|---|------|--------|------|--------|------|
| 1 | 입력이 간편하면 지속 사용 | 그래도 귀찮아서 3일 만에 포기 | 최근 기록 재사용, 자동완성 | 3일 연속 미기록 | 입력 단계 더 축소 |
| 2 | 음식 DB API 품질이 좋을 것 | 한국 음식 데이터 부족 | 자주 먹는 한국 음식 직접 등록 | 검색 실패율 30% 초과 | 사용자 직접 입력 + DB 확장 |
| 3 | 로컬 저장으로 충분 | 브라우저 캐시 삭제 시 데이터 유실 | IndexedDB는 캐시 삭제에 포함 안 됨 안내 | 데이터 유실 발생 | 내보내기(JSON) 기능 추가 |
| 4 | PWA로 모바일 경험 충분 | 네이티브 앱 대비 UX 열등 | 홈 화면 추가 유도, 오프라인 지원 | 사용 불편 피드백 | React Native 전환 검토 |
| 5 | 수치만 보여주면 동기 유지 | 숫자만으로는 지루해서 포기 | 깔끔한 시각화로 보는 재미 | 2주 후 사용 빈도 감소 | 간단한 달성 피드백 추가 |

## 7. 해결 가설과 실험

### 7.1 가설
> 만약 음식 이름만 검색하면 칼로리가 자동으로 입력되게 하면, 매일 3끼 식단 기록을 지속할 것이다. 왜냐하면 기록 포기의 가장 큰 이유가 "귀찮은 입력"이기 때문이다.

### 7.2 실험 루프

```
실험 → 관측 → 학습 → 다음 가설
```

| 단계 | 내용 |
|------|------|
| 실험 | MVP를 만들어 1주일간 직접 매일 사용 |
| 관측 | 일일 기록 횟수, 기록 소요 시간, 포기 시점 |
| 학습 | 3끼 기록률 50% 이상이면 성공, 미만이면 입력 방식 재설계 |
| 다음 | 성공 시 FEAT-2(상세 칼로리) 추가, 실패 시 입력 UX 개선 |
